### 🔴 BUG #1 — 严重：普通用户可访问上机物料 API

| 属性 | 详情 |
|------|------|
| **严重程度** | 🔴 高 (安全漏洞) |

| **文件** | `backend/src/main/java/com/metal/controller/MachineMaterialController.java` |
| **根因** | 控制器所有方法均**缺少** `ServiceHelper.requireAdmin()` 调用 |
| **现象** | 普通用户 (test/test123) 直接请求 `/api/machine-material?page=1&pageSize=5` 返回 `code=200` + 完整数据列表 |
| **期望** | 返回 `code=500, msg="无权限：仅管理员可操作"` |
| **修复方案** | 在 `MachineMaterialController` 的所有方法开头添加 `ServiceHelper.requireAdmin()`，参照 `OperationLogController` 的实现方式 |

**对比参考** (正确的实现):
```java
// OperationLogController.java — 正确做了权限检查
@GetMapping
public Result<PageResult<OperationLog>> query(...) {
    ServiceHelper.requireAdmin();  // ← 这行
    return Result.ok(service.query(...));
}

// MachineMaterialController.java — 缺少此检查
@GetMapping
public Result<PageResult<MachineMaterial>> query(...) {
    // 缺少 ServiceHelper.requireAdmin();
    return Result.ok(service.query(...));
}
```

### 🔴 BUG #2 — 严重：超比统计 ratio 字段存储错误（15.0 而非 0.15）

| 属性 | 详情 |
|------|------|
| **严重程度** | 🔴 高 (数据完整性问题) |
| **测试项** | 5.1 百分比存储 |
| **文件** | `backend/src/main/java/com/metal/service/DeliveryStatsService.java:264` |
| **根因** | `applyCalculations()` 方法**不包含** ratio `/100` 转换逻辑。ratio `/100` 转换仅在 Excel 导入时执行（第160-166行），而 `create()` 和 `update()` 通过 API 调用时直接使用原始值 |
| **现象** | 前端输入 `15` (表示15%)，API 传入 `ratio=15`，数据库存储 `15.0` 而非 `0.15` |
| **期望** | 数据库存储 `0.15`，列表显示 `15%`，编辑框回显 `15` |

**代码分析**:
```
控制器 parseStats() → ratio=15 → BigDecimal(15)
    ↓
Service create() → applyCalculations(record)  ← 无 /100 转换！
    ↓
mapper.insert() → DB 存 15.0

仅 Excel import 有转换逻辑（第160-166行）:
if (r.compareTo(BigDecimal.ONE) > 0) {
    data.setRatio(r.divide(BigDecimal.valueOf(100), ...));
}
```

**修复方案**: 在 `applyCalculations()` 方法开头添加 ratio 转换，或在 `parseStats()` 中处理：
```java
// 方案1: 在 applyCalculations() 开头添加
private void applyCalculations(DeliveryStats record) {
    // 百分比转换：前端传 15 → 存 0.15
    if (record.getRatio() != null && record.getRatio().compareTo(BigDecimal.ONE) > 0) {
        record.setRatio(record.getRatio().divide(BigDecimal.valueOf(100), 4, RoundingMode.HALF_UP));
    }
    // ... 现有逻辑
}

// 方案2: 在 parseStats() 中处理
if (body.get("ratio") != null) {
    BigDecimal r = new BigDecimal(body.get("ratio").toString());
    if (r.compareTo(BigDecimal.ONE) > 0) {
        r = r.divide(BigDecimal.valueOf(100), 4, RoundingMode.HALF_UP);
    }
    s.setRatio(r);
}
```

### 🔴 BUG #3 — 严重：结算机台 ratio 字段存储错误（25.0 而非 0.25）

| 属性 | 详情 |
|------|------|
| **严重程度** | 🔴 高 (数据完整性问题) |
| **测试项** | 6.1 百分比存储 |
| **文件** | `backend/src/main/java/com/metal/service/SettlementMachineService.java:56` |
| **根因** | 与 Bug #2 相同 — `SettlementMachineService.create()` 和 `update()` 不转换 ratio。转换逻辑仅在 Excel 导入时存在（第133-137行） |
| **现象** | 前端输入 `25` (表示25%)，数据库存储 `25.0` 而非 `0.25` |
| **期望** | 数据库存储 `0.25` |

**修复方案**: 在 `SettlementMachineService.create()` 和 `update()` 方法中，调用 mapper 之前添加 ratio 转换：
```java
@Transactional
public SettlementMachine create(SettlementMachine record) {
    // 添加百分比转换
    if (record.getRatio() != null && record.getRatio().compareTo(BigDecimal.ONE) > 0) {
        record.setRatio(record.getRatio().divide(BigDecimal.valueOf(100), 4, RoundingMode.HALF_UP));
    }
    // ... 现有逻辑
}
```

## 七、代码审查备注

以下问题虽不影响功能但值得关注：

1. **SecurityConfig.java:27** — `requestMatchers("/api/**").permitAll()` 放开所有 API，权限完全依赖 Interceptor + Controller/Service 层的显式检查。建议考虑在 SecurityConfig 层面对 `/api/admin/**` 做角色限制作为纵深防御。

2. **VoiceParseService** — 纯本地关键词锚点解析，不调用外部API。返回结构 `{"fields": {...}, "filledCount": N}` 正确。自动化测试脚本中字段计数逻辑有误（计了顶层而非 `fields` 内层），已确认实际填充字段数正确，不存在Bug。

3. **MachineCount 百分比语义** — 该表的 `ratioPct` 字段存 0~100 范围（与其他表的 0~1 不同），已在测试清单中注明，不需要修改，但需确保文档清晰。
